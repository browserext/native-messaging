Various applications of Browser Extensions need a mechanism to communicate with local client native processes outside of the browser environment. This specification proposes such a mechanism.

Introduction

The Browser Extensions Community Group's goal is to make browser extension code much more interoperable across browsers by specifying common extension interfaces and well-defined browser behavior.

This specification defines the interfaces and behavior which enables extensions to communicate with local client native processes outside of the browser environment.

It is not a goal of this specification to define:

  • The mechanism a browser would use to communicate with the local client native process (e.g. stdio)
  • The registration specifics of how a browser would locate the local client native process app, service, or binary executable
  • The specifics of how a browser would manage the lifetime of local client native process

Native Application Registration

Extensions using the native messaging APIs identify a native application with which they wish to communciate using a string identifier. A browser that implements native messaging must also define a method for mapping a string identifier to a native application. This method MUST allow the browser to do the following:

  • Locate the associated native component on the local client
  • Start the native process when needed
  • Exchange messages containing JSON serializable data

The method for identifying and communicating with native processes MAY also do the following:

  • Establish a limit on the size of messages sent both to and from the native application, to avoid resource exhaustion
  • Define a method for exchanging additional types of messages (i.e., data that is not JSON serializable)
  • Impose access control, allowing the mapping for a particular native application to be restricted to a set of specified extensions

Each browser implementation may have different mechanisms for registration, message filtering and lifetime management of the local client native process.

Core APIs

The goal for these APIs is to provide functionality that extension authors need for message-passing between extensions and local client native process outside of the browser. Methods are provided for both establishing long-lived connection ports and for sending individual "one-shot" messages.

The runtime object

Overview

In addition to the base functionality of the browser.runtime object defined in the [[!browserext]] Browser Extensions specification, these additional APIs provide functionaly for Native Messaging.

Manifest

The "nativeMessaging" permission is required to use these additional APIs. 

{
    "permissions": [                               // Required
        "nativeMessaging"                          // Required
    ]                                              // Required   
}

Context

This API is available only in background pages, a subset of the Window context.

WebIDL Definition

A description of the special WebIDL syntax considerations for browser extensions are defined elsewhere in the [[!browserext]] Browser Extensions specification. 

interface BrowserExtRuntimePort {
    void disconnect();
    DOMString name;
    Error? error;
    BrowserExtEvent onDisconnect;
    BrowserExtEvent onMessage;
    void postMessage(object msg);
    MessageSender sender;
};
[NoInterfaceObject]
partial interface BrowserExtBrowserRuntimeNativeMessaging {
    BrowserExtRuntimePort connectNative(DOMString nativeServiceName);
    Promise<object> sendNativeMessage(DOMString nativeServiceName, object message);
    attribute BrowserExtRuntimePort Port;
};
[NoInterfaceObject, Exposed=Window,Ck_Any_Prm_nativeMessaging_]
interface BrowserExtRuntimeNativeMessagingAPI {
    readonly attribute BrowserExtRuntimeNativeMessaging runtime; 
};
Browser implements BrowserExtRuntimeNativeMessagingAPI;

Description of functions

connectNative()
This function attempts to establish a long-lived connection with the application identified by nativeServiceName. The process by which the requested native application is identified and communication is established is outside of the scope of this document, as detailed above. A BrowserExtRuntimePort object is immediately returned. If the requested application does not exist or communication cannot be established for any reason, the error property of the returned object will be non-null.
sendNativeMessage()
This function attemps to establish a temporary connection with the application identified by nativeServiceName and transmit the provided message to it. If communication with the specified application cannot be established, the returned Promise is rejected with an appropriate error. If communication is established, the returned Promise is resolved with the contents of the first message sent from the native application to the browser. After the returned Promise is settled, the logical connection established by this function is longer accessible, an extension must call connectNative() or sendNativeMessage() again to conduct further communication. Though note that the logical connection need not be tied to the lifetime of an individual process on the client, different browsers may choose to manage processes differently.
